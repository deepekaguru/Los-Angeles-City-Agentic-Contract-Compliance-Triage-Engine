# Logical Structure & System Architecture

## 1. System Ecosystem Overview
The application follows an Event-Driven Multi-Agent Architecture using a decoupled Frontend, a relational database core, and an Agentic Orchestration layer communicating via structured protocol tools.

## 2. Relational Database Schema
The database layer enforces relational integrity across three core tables:

### `contracts`
* `contract_id` (TEXT, PRIMARY KEY): Unique identifier for the city contract.
* `vendor_name` (TEXT): Name of the vendor.
* `max_budget` (REAL): Maximum allowable spend.
* `approved_category` (TEXT): Scope of work permitted (e.g., "IT", "Logistics").

### `transactions`
* `transaction_id` (TEXT, PRIMARY KEY): Unique identifier for the invoice line-item.
* `contract_id` (TEXT, FOREIGN KEY references `contracts`): Linked contract.
* `amount` (REAL): Transaction cost.
* `category` (TEXT): Line-item category.
* `timestamp` (TEXT): ISO-8601 timestamp.

### `compliance_audit_logs`
* `log_id` (INTEGER, PRIMARY KEY AUTOINCREMENT)
* `transaction_id` (TEXT, FOREIGN KEY references `transactions`)
* `status` (TEXT): 'PASSED', 'FLAGGED', or 'CRITICAL'.
* `reasoning_summary` (TEXT): Explanatory text generated by the agent detailing the audit conclusion.

## 3. Agentic Workflow Sequence
1. **Trigger:** The UI passes a raw payload of new transactions to the Agentic Orchestrator.
2. **Analysis:** The Orchestrator invokes the **Compliance Auditor Agent**, passing the transaction details and fetching corresponding contract limits via database tools.
3. **Evaluation:** If a transaction exceeds the contract's `max_budget` or violates the `approved_category`, the payload is routed to the **Triage Agent**.
4. **Persist & Present:** The agent writes the structured evaluation directly into `compliance_audit_logs` via an MCP SQL tool call. The frontend refreshes to reflect the newly audited pipeline status.
